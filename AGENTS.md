# AGENTS.md

## 이 파일의 역할
이 파일은 이 저장소에서 작업하는 AI 에이전트의 운영 규칙 정본이다.
구현보다 안전한 반복 개발과 사람 승인 기반 운영을 우선한다.

---

## 작업 시작 규칙

작업 시작 시 먼저 현재 작업의 성격을 판단한다.

### 1) 구조 / 아키텍처 / 코드베이스 이해 질문
아래 순서로 확인한다.
1. `graphify-out/GRAPH_REPORT.md`
2. 필요하면 `graphify-out/wiki/index.md`
3. 그다음 관련 원본 코드 / 문서

### 2) 큰 작업
아래 문서를 먼저 읽는다.
- `docs/PRODUCT.md`
- `docs/ARCHITECTURE.md`
- `docs/CURRENT_SCOPE.md`
- `docs/TASKS.md`

### 3) 작은 작업
- 관련 파일과 관련 문서만 최소 범위로 읽는다.
- 구조 이해가 필요하면 먼저 `graphify-out/GRAPH_REPORT.md`를 본다.
- 범위가 커지면 큰 작업 절차로 전환한다.

위 문서와 실제 코드 상태가 충돌하면:
1. 실제 저장소 상태를 먼저 확인한다.
2. 충돌 내용을 명시적으로 보고한다.
3. 승인 없이 구현 방향을 밀어붙이지 않는다.

---

## 작업 원칙
- 승인된 범위를 넘어서 구현하지 않는다.
- 큰 작업은 바로 코딩하지 말고 먼저 계획을 세운다.
- 작업은 가능한 작은 단위로 쪼갠다.
- 변경은 작고 명시적이고 되돌리기 쉽게 유지한다.
- 저장소 근거 없는 기능, 역사, 설계를 만들어내지 않는다.
- 숨은 가정보다 명시적 가정을 우선한다.
- 의미 있는 변경 뒤에는 관련 문서를 동기화한다.
- Graphify는 탐색 우선순위를 정하는 지도처럼 사용한다.
- 구현 변경 전에는 반드시 실제 코드와 문서로 근거를 다시 확인한다.
- `INFERRED`, `AMBIGUOUS` 관계는 검증 전 사실로 단정하지 않는다.

---

## 큰 작업의 기준
아래 중 하나라도 해당하면 큰 작업으로 본다.
- 아키텍처나 모듈 경계를 바꾼다.
- tool-calling 동작이나 실행 흐름에 영향을 준다.
- 의존성을 추가하거나 제거한다.
- 공개 인터페이스를 바꾼다.
- persistence / logging / evaluation / safety 로직을 바꾼다.
- 3개 초과 파일을 실질적으로 수정한다.
- 수용 기준이 불명확하다.
- CLI와 Web 양쪽에 동시에 영향이 갈 수 있다.

저장소 예시:
- `tool_registry.py` 의 노출 툴 변경
- `agent.py` 의 planning / execution 흐름 변경
- `app/api` 와 `frontend/src/api.ts` 를 함께 바꾸는 작업
- 상태 저장 구조나 trace 형식을 바꾸는 작업

---

## 큰 작업에서의 행동
큰 작업이면 아래 순서를 따른다.
1. 바로 구현하지 않는다.
2. 현재 코드와 문서를 분석한다.
3. 필요하면 먼저 `graphify-out/GRAPH_REPORT.md`로 구조를 파악한다.
4. 단계별 계획을 정리한다.
5. 가정, 리스크, 검증 방법을 명시한다.
6. 사람 승인 전에는 구현에 들어가지 않는다.

---

## 작은 작업에서의 행동
작은 작업이고 승인된 범위 안이라면 아래 순서를 따른다.
1. 최소 범위만 수정한다.
2. 필요한 최소 검증만 수행한다.
3. 관련 문서를 갱신한다.
4. `docs/CHANGELOG.md` 에 실제 변경만 기록한다.
5. 상태가 바뀌었다면 `docs/CURRENT_SCOPE.md` 와 `docs/TASKS.md` 초안을 갱신한다.
6. 구조적 의미가 있는 변경이면 작업 끝에 Graphify 갱신 필요 여부를 판단한다.

---

## 완료 기준

### 문서 작업 완료
문서 작업은 아래를 모두 만족해야 완료로 본다.
- 필수 문서가 생성 또는 갱신되었다.
- 문서 내용이 실제 저장소 근거와 맞는다.
- 초안과 확정 사실이 구분되어 있다.
- 관련 문서 간 역할과 정본 위치가 모순되지 않는다.
- `docs/CHANGELOG.md` 에 이번 실제 작업이 기록되었다.

### 코드 작업 완료
코드 작업은 아래를 모두 만족해야 완료로 본다.
- 승인된 범위 안에서만 변경되었다.
- 변경이 작고 검토 가능한 diff 로 남아 있다.
- 최소 검증이 수행되었거나, 못 했다면 이유가 보고되었다.
- 관련 문서가 실제 상태에 맞게 갱신되었다.
- `docs/CHANGELOG.md` 에 실제 변경이 기록되었다.
- 구조적 변경이 있었다면 Graphify 갱신 필요 여부가 판단되었다.

---

## 사람 승인이 필요한 경우
아래 경우에는 사람 승인 없이 진행하지 않는다.
- 큰 작업 시작 전
- 실행 흐름, 툴 노출, 아키텍처 경계 변경
- 공개 인터페이스 변경
- persistence / logging / evaluation / safety 로직 변경
- 의존성 추가 또는 삭제
- destructive action
- 수용 기준이 불명확한 경우
- 현재 문서와 실제 코드가 크게 충돌해 방향 결정을 새로 내려야 하는 경우

---

## 모호할 때의 행동
- 범위를 줄여서 확실한 부분만 정리한다.
- 필요한 가정을 명시한다.
- 구현보다 문서화와 질문 정리를 우선한다.
- 승인 없이 큰 설계 결정을 확정하지 않는다.

---

## 가정 보고 방식
가정은 숨기지 않고 아래 기준으로 보고한다.
- 무엇을 가정했는지
- 왜 그 가정을 했는지
- 어떤 저장소 근거가 있었는지
- 어떤 부분은 사람 확인이 필요한지

문서에는 가능하면 `가정`, `초안`, `사람 확인 필요` 같은 표현으로 드러나게 쓴다.

## subagent 사용 규칙
- subagent는 기본값이 아니다.
- review, test review, failure/log analysis, architecture critique 같이 분리 가능한 분석 작업에 한해 고려한다.
- 현재 실행 환경이 허용하고, 사람 승인 또는 명시적 위임이 있을 때만 사용한다.
- 구현 본작업을 subagent에 넘기기 전에 주 작업의 책임 범위를 먼저 분리한다.
- Graphify 생성이나 대규모 semantic 추출처럼 비용이 큰 작업에서는 subagent 사용을 기본값으로 두지 않는다.

## 종료 전 문서 동기화 확인
작업 종료 전 아래를 반드시 확인한다.
- `docs/PRODUCT.md`, `docs/ARCHITECTURE.md`, `docs/CURRENT_SCOPE.md`, `docs/TASKS.md` 가 현재 상태와 충돌하지 않는지
- 새로 발견된 known issue 가 `CURRENT_SCOPE.md` 또는 `TASKS.md` 에 반영되었는지
- 정본 위치가 README와 충돌하지 않는지
- `docs/CHANGELOG.md` 에 이번 실제 변경이 기록되었는지
- 구조적으로 의미 있는 변경이었다면 Graphify 갱신이 필요한지 확인했는지

## Graphify 사용 규칙

이 프로젝트에는 `graphify-out/` 아래에 Graphify 지식 그래프가 있다.

규칙:
- 아키텍처, 모듈 경계, 의존 흐름, 코드베이스 구조 질문에 답하기 전에는 먼저 `graphify-out/GRAPH_REPORT.md`를 읽는다.
- `graphify-out/wiki/index.md`가 있으면, 많은 원본 파일을 바로 열기 전에 그 문서를 탐색 레이어로 사용한다.
- Graphify는 허브 노드, 커뮤니티 경계, 우선 확인할 파일 후보를 찾는 데 먼저 사용한다.
- Graphify는 지도이지 최종 증거가 아니다.
- `INFERRED`, `AMBIGUOUS` 관계는 실제 코드나 문서로 검증하기 전까지 구현 근거로 사용하지 않는다.
- 작은 수정마다 Graphify를 다시 빌드하지 않는다.
- 아래 중 하나라도 해당할 때만 Graphify 갱신을 고려한다:
  - 모듈 경계가 바뀌었다
  - 실행 흐름이 바뀌었다
  - 공개 인터페이스가 바뀌었다
  - 서로 관련된 여러 파일이 의미 있게 함께 바뀌었다
  - 최신 그래프가 필요한 구조 질문에 답해야 한다
- Graphify 갱신이 필요하면, 이 프로젝트에서 사용하는 현재 Graphify 갱신 명령으로 반영한다.
- 가능하면 생성 산출물과 캐시 디렉토리는 Graphify 범위에서 제외한다.

## Graphify 갱신 정책

Graphify는 선택적으로 갱신한다.

다음 경우에는 갱신을 권장한다.
- 구조적으로 의미 있는 변경 뒤
- 그래프가 낡았을 수 있는 상태에서 새로운 아키텍처 질문에 답하기 전
- 코드베이스 이해에 영향을 주는 문서 / 노트가 추가된 뒤
- 현재 구조를 크게 요약해서 넘기기 전

다음 경우에는 보통 갱신하지 않는다.
- 한 줄 수정
- 작은 지역 리팩터링
- 구조 영향이 없는 텍스트 수정
- 1~2개 파일만 보면 바로 답할 수 있는 작업
